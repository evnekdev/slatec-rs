# Documentation audit report

## Scope and snapshot

This audit reviews the collection generated by Prompts 01–09 on the current default-branch documentation snapshot observed on 16 July 2026. It evaluates completeness, consistency, citation practice, terminology, metadata conventions, domain overlap and readiness for bulk routine extraction.

No existing technical page was silently rewritten. The ZIP adds only the five Prompt 10 deliverables.

## Method

The review used:

- the prompt-required file manifests;
- repository code search and direct retrieval of representative current files;
- cross-comparison of source, architecture, taxonomy, domain, routine, dependency and Rust documents;
- validation history recorded by earlier prompts for TOML files;
- manual inspection of internal relative-link patterns;
- comparison of metadata IDs and terminology where visible.

A complete local checkout and executable Markdown link checker were unavailable during this pass. Therefore link and inventory findings are classified as **manual/preliminary** and a repository-local CI audit remains mandatory.

## Terminology consistency

### Consistent terms

The collection generally uses:

- **user-callable** for documented primary interfaces;
- **subsidiary** for implementation/support routines;
- **package** only when historical package evidence exists;
- **family** for weaker or partial provenance;
- **domain** for functional classification;
- **native component** for a linkable implementation unit;
- **raw crate** and **safe crate** for Rust API layers.

### Inconsistencies to normalize

**Finding TERM-1, P2:** “public routine” occasionally appears as a synonym for “user-callable.” Prefer `user-callable`, because link visibility is not the same concept.

**Finding TERM-2, P2:** evidence vocabularies are not fully unified. The routine schema permits `verified`, `derived`, `inferred`, `unverified`, `conflicting`; source records use retrieval-specific values such as `linked-unverified-content`, while package confidence includes `high`, `medium`, `low`, `verified`. These are valid separate dimensions, but documentation should define them centrally and prevent accidental interchange.

**Finding TERM-3, P3:** “package”, “collection”, “family” and “subsystem” are used carefully but no machine-readable provenance-kind enum document currently governs all metadata.

**Finding SID-2, P2:** distinguish human Markdown link labels from source IDs. Labels such as ``[`netlib-quadpack`](URL)`` look like IDs but are not automatically validated against `metadata/sources.toml`.

**Finding SID-3, P2:** ensure every source ID is unique and stable when URLs or titles are corrected.

## Citation quality

### Strengths

- primary sources are consistently preferred;
- routine pages cite official source/prologue URLs;
- uncertain package boundaries are explicitly qualified;
- modern comparisons are identified as interpretation;
- Rust design pages primarily cite collected internal evidence.

### Weaknesses

**Finding CIT-1, P1:** broad “Sources” sections do not always provide a locator for each table row or detailed claim. Bulk-generated pages should record source line ranges, prologue fields or source hashes.

**Finding CIT-2, P1:** several package/family claims rely on index-level evidence without source-diff verification. They must remain medium/high confidence rather than exact routine ownership.

**Finding CIT-3, P2:** modern-library comparison paragraphs need stable official documentation references and retrieval dates in the source register if they remain part of the long-term knowledge base.

**Finding CIT-4, P2:** manually written routine family tables should be regenerated from routine metadata after full extraction to prevent drift.

## Duplicate and contradictory claims

### No critical architectural contradiction found

The following statements are compatible:

- SLATEC is distributed as a broad common library;
- Netlib later moved subsets into separate directories;
- packages can have identifiable provenance;
- safe Rust domains need not match historical packages;
- one native owner is preferred initially;
- later native splitting is conditional on dependency evidence.

### Tensions requiring explicit resolution

**Finding CON-1, P1:** some domain pages list routine/package membership more confidently than package metadata can currently prove. Treat representative family tables as orientation until source comparison populates exact membership.

**Finding CON-2, P1:** transforms documentation discusses predominantly single-precision FFTPACK while future crate planning sometimes speaks generically of precision variants. Full inventory must decide actual S/D/C coverage before generic APIs.

**Finding CON-3, P1:** statistics discusses historical RNG routines without verified algorithm/state analysis. Do not expose them safely until source inspection confirms state, period and thread behavior.

**Finding CON-4, P2:** “one native core with logical raw crates” and “initial monolithic `slatec-sys`” use slightly different names for the same phase. Standardize the bootstrap name before implementation.

**Finding CON-5, P2:** domain metadata dependency layers are intentionally conceptual, while dependency schema defines source-level edges. Documentation should keep these two graph types visibly distinct in generated diagrams.

## Domain overlap review

Overlaps are generally recognized correctly:

- special functions ↔ probability/statistics;
- interpolation ↔ approximation ↔ least squares;
- nonlinear equations ↔ optimization;
- transforms → FISHPACK/PDE;
- dense kernels → sparse/ODE/optimization;
- spline representations → integration;
- runtime support → all domains.

**Finding DOM-1, P2:** define explicit cross-listing metadata rather than duplicating routine records between domains.

**Finding DOM-2, P2:** decide whether numerical differentiation remains within quadrature or becomes a separate subdomain.

**Finding DOM-3, P3:** decide whether statistics deserves a public crate after exact routine inventory.
