# `0.1.0` publication guide

This guide prepares the first coordinated release; it does not authorize a
publish, tag, or GitHub release. The selected commit must have passed the
target-specific CI workflows and the manual release gates below.

## Release scope

All six published packages use version `0.1.0`, Rust edition 2024, and MSRV
Rust 1.85:

| Package | Role | Publication order |
| --- | --- | ---: |
| `slatec-bundled-x86_64-pc-windows-gnu` | Windows GNU carrier metadata, archive, and runtime closure | 1 |
| `slatec-bundled-x86_64-unknown-linux-gnu` | Linux GNU carrier metadata, archive, and runtime closure | 2 |
| `slatec-sys` | provider-neutral raw declarations | 3 |
| `slatec-core` | provider-neutral support contracts | 4 |
| `slatec-src` | provider choice and native linking | 5 |
| `slatec` | selective safe facade | 6 |

The publication order is intentionally serial even where the package graph
permits parallel uploads. It gives downstream resolution one unambiguous
carrier and raw-declaration prerequisite at a time.

## Supported bundled targets and families

The `bundled` provider is supported only on the following targets. All ten
listed families are present in each carrier; all other families require a
deliberate `source-build`, `system`, or `external-backend` selection.

| Target | Bundled families | Native archive | Runtime closure |
| --- | ---: | ---: | ---: |
| `x86_64-pc-windows-gnu` | 10 | 385,314 bytes | `libgfortran` and `libquadmath` static archives |
| `x86_64-unknown-linux-gnu` | 10 | 567,600 bytes | `libgfortran` and `libquadmath` static archives |

Families: `roots-scalar`, `special-airy`, `special-bessel`, `special-beta`,
`special-elementary`, `special-error`, `special-gamma`,
`special-integrals`, `special-polynomials`, and
`special-scalar-expanded`.

The exact archive and runtime hashes, member evidence, package sizes, and
licence-file extraction are in
[`generated/release-readiness/final-independent-audit.json`](../../generated/release-readiness/final-independent-audit.json)
and [`package-size-audit.json`](../../generated/release-readiness/package-size-audit.json).

## Providers and limitations

`slatec-sys` is declaration-only: it neither selects nor builds a native
provider. `slatec-src` selects exactly one provider for native family use:

- `bundled` uses an exact target carrier and rejects a family without a
  redistributable verified archive.
- `source-build` is offline-only and needs a separately acquired,
  checksum-verified source cache plus a matching GNU Fortran toolchain.
- `system` requires the caller to provide a validated static SLATEC archive
  and static GNU runtime archives.
- `external-backend` leaves native provisioning to the integrating program.

The safe API is intentionally selective. It does not wrap all raw routines,
and some raw contracts retain explicit unavailable semantic fields when the
source evidence is insufficient. Other targets and provider/toolchain ABIs are
not bundled support claims.

## Licences, provenance, and relinking

Rust source is licensed under `MIT OR Apache-2.0`. Carrier packages include
their exact GNU runtime notices: GPL-3.0, LGPL-2.1-or-later, the GCC Runtime
Library Exception 3.1, and `SOURCES-AND-RELINKING.md`. Each carrier archive is
checksum-verified against a generated receipt. Downloaded SLATEC source caches
are not redistributed; source-rights review remains an explicit scope
boundary.

## Mandatory evidence before publication

Run the independent audit and examine all generated evidence:

```text
cargo run -p slatec-tools --bin slatec-corpus -- validate-final-release-audit --offline
cargo run -p slatec-tools --bin slatec-corpus -- validate-registry-simulation --offline
cargo run -p slatec-tools --bin slatec-corpus -- validate-package-contents --offline
```

The committed all-target registry evidence is generated from Windows with WSL:
set `SLATEC_REGISTRY_TARGET=both` for that full rehearsal. Target-specific CI
uses `linux` or `windows` and writes its transient observation under `target/`
so a single-platform worker cannot overwrite the all-target evidence.

Observe successful `Release consumers / Linux GNU registry-only consumer` and
`Release consumers / Windows GNU registry-only consumer` workflow runs for the
selected commit. The workflows use only packaged-directory-source consumers,
set `SLATEC_GFORTRAN` to a deliberately invalid value, and exercise a
one-family plus all-bundled-family program. They do not claim a source-build
test.

Immediately before publication, use an up-to-date advisory database to run
the local policy command (for example `cargo audit`), then record the result.
Use `cargo public-api` snapshots for `slatec`, `slatec-core`, `slatec-sys`, and
both carriers; use `cargo semver-checks` where a suitable published baseline
is available. There is no prior published `0.1.x` API to compare at this first
release, so these commands establish external snapshots rather than inventing
a historical compatibility result.

## Publication and post-publication operations

Run [`scripts/release-0.1.0-dry-run.ps1`](../../scripts/release-0.1.0-dry-run.ps1)
first. After every prerequisite package is visible in the index, use the
guarded [`scripts/release-0.1.0-publish-template.ps1`](../../scripts/release-0.1.0-publish-template.ps1)
from the selected release commit.

After publication, verify each package with `cargo info <crate>@0.1.0`, wait
for docs.rs build results, download both carrier archives and compare their
SHA-256 values with the audit, then run clean Windows GNU and Linux GNU
one-family/all-family consumers. Finally re-check the dependency graph, tag
the exact commit as `v0.1.0`, and create the matching GitHub release only after
all package, documentation, and consumer evidence is recorded.
